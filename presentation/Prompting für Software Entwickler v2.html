Das wichtigste Mental Model

KI ist kein Zauberstab

KI kann deine Gedanken nicht lesen — nur deine Worte. Das ist der Hebel.

Was Devs oft denken

„Ich schreibe einen kurzen Prompt, und KI versteht automatisch, was ich wirklich brauche."

Das funktioniert nur, wenn die Aufgabe trivial ist.

Was wirklich passiert

KI liest deine exakten Worte und antwortet auf das, was du geschrieben hast — nicht auf das, was du denkst.

„Mach das schneller" → KI kennt das Problem nicht.

„Mach Payload kleiner" → KI weiß nicht, welche Felder optional sind.

Merke: KIs reagieren nur auf deinen Input — sie lesen keine Gedanken und können nicht zaubern.

Garbage In, Garbage Out (GiGo)

Vager Prompt

„Fix den Bug."

↓ KI rät: welcher Bug? welche Datei? Refactor? Neue Architektur?

Präziser Prompt

„In orderCalculator.js gibt calculateTotal() bei leerem lineItems-Array 4,99 zurück — korrekt wäre 0. Der Fehler liegt in Zeile 15–17. Passe nur diesen Fall an, Signatur und übrige Rabatt-/Versandlogik bleiben unverändert."

↓ Gezielter Fix — kein Raten, kein Over-Engineering

💡 Die KI spiegelt die Qualität deiner Eingabe.

Context is King

Was muss die KI wirklich verstehen, um dir zu helfen?

Ziel
Was soll am Ende herauskommen?

z. B. 100 % Branch-Coverage, vollständige Edge-Case-Liste, lauffähige Test-Suite

Code
Welche Datei/n? Welcher Modul?

z. B. orderCalculator.js, Funktion calculateTotal(), Zeilen 14–33

Problem
Was funktioniert nicht? Was ist das Symptom?

z. B. leeres Array → 4,99 statt 0; run-tests.bat schlägt fehl

Grenzen
Performance? Legacy? Tests? API-Constraints?

z. B. nur Node --test, kein Jest; Laufzeit < 50 ms; .NET 4.8

Stabil
Was darf sich NICHT ändern?

z. B. Funktionssignatur, öffentliche API, bestehende Tests bleiben grün

Kontext
Was ist der größere Zusammenhang?

z. B. Checkout-Flow, Versandfrei ab 50 €, Rabatt vor Versand

CRAFT: Struktur für präzise Prompts

Jeder Baustein beantwortet eine Frage — mit konkretem Inhalt statt Labels.

C

Context

Welche Dateien, welcher Stand, welche Randbedingungen?

z. B. orderCalculator.js, calculateTotal(), lineItems, Versandfrei ab 50 €

R

Role

Aus welcher Perspektive soll die KI arbeiten?

z. B. Test-Autor, sorgfältiger Backend-Dev, Code-Reviewer

A

Action

Was soll die KI konkret tun?

z. B. Unit Tests für calculateTotal() schreiben, alle Edge Cases abdecken

F

Format

Wie soll die Antwort aussehen?

z. B. Node --test, AAA-Pattern, nur Code für orderCalculator.test.js

T

Constraints

Was darf NICHT passieren?

z. B. Funktionssignatur unverändert, kein Jest, bestehende Tests bleiben grün, kein Refactor

01 Plan-First

Größere Features: erst planen, dann coden

Wann?

Mehrere Dateien, unklarer Lösungsweg, komplexes Feature

Wie?

  1. Plan anfragen
  2. Du prüfst
  3. Code Step by Step

Beispiel-Prompt

„Plan für Feature X in Datei Y. 8 nummerierte Schritte. Kein Code. Risiken?"

ZwickRoell
02 Test-Driven

Tests zuerst, dann Implementierung

Wann?

Neue Logik, wo Tests fehlen, Regression vermeiden

Wie?

  1. Tests (happy, edge, error)
  2. Implementierung
  3. Grün = fertig

Beispiel-Prompt

„Tests für calculateTotal. AAA-Format. Happy, leer, negativ. node:test. Kein Code außer Tests."

ZwickRoell
03 Review-First

Legacy-Code: erst reviewen, dann ändern

Wann?

Fremder Code, hohes Risiko, unklare Intent

Wie?

  1. Review (Risiken, Tests, Intent)
  2. Du fragst nach
  3. Dann gezielte Änderung

Beispiel-Prompt

„Review Code-Block. Risiken? Fehlende Tests? Intent? Kein Code ändern."

ZwickRoell
04 Rubber-Duck

Bugs: Symptom beschreiben, Hypothesen sammeln

Wann?

Bug, flaky Test, unklares Verhalten, merkwürdiges Symptom

Wie?

  1. Symptom + Kontext
  2. Top-Hypothesen
  3. Prüf-Steps

Beispiel-Prompt

„calculateTotal liefert 0 statt Exception. Letzte Änderung: Rabatt. Top-3 Hypothesen + Prüf-Steps. Kein Code."

ZwickRoell
05 Explain-Before-Change

Verständnis vor der Änderung

Wann?

Unbekannter Code, vor Änderung Klarheit nötig

Wie?

  1. Erklärung anfragen
  2. Du validierst
  3. Dann erst Änderung

Beispiel-Prompt

„Erkläre calculateTotal in 5 Bullets: Input, Output, Edge Cases, Dependencies. Kein Code."

ZwickRoell
06 Spec-Driven

Erst die Spec, dann der Code

Wann?

Klar definierte Features mit testbaren Anforderungen

Wie?

  1. Akzeptanzkriterien schreiben
  2. KI implementiert danach
  3. Output gegen Spec prüfen

Beispiel-Prompt

„Implementiere nach dieser Spec: Gegeben leerer Warenkorb, wenn Artikel hinzugefügt, dann Summe = Preis. Erfülle alle 3 Kriterien."

ZwickRoell
07 Few-Shot

Zeig der KI ein Beispiel aus deinem Code

Wann?

Konsistenz mit bestehenden Patterns ist wichtig

Wie?

  1. Bestehendes Pattern als Beispiel
  2. „Mach es genauso"
  3. Stil bleibt konsistent

Beispiel-Prompt

„Schreib den neuen PaymentService genau im Stil von UserService.js — gleiche Struktur, gleiche Fehlerbehandlung."

ZwickRoell
08 Red-Team

Lass die KI ihren eigenen Code angreifen

Wann?

Vor dem Merge, bei kritischem oder sicherheitsrelevantem Code

Wie?

  1. Lösung generieren lassen
  2. „Finde 3 Schwächen"
  3. Fixes bewerten & übernehmen

Beispiel-Prompt

„Finde 3 Schwachstellen in dieser Lösung: Security, Edge Cases, Performance. Sei kritisch, beschönige nichts."

🧑‍💻 ÜBUNG 3

2–3 Techniken live ausprobieren

15 Min Arbeit · 3 Min Debrief · Paar oder Einzel · Anknüpfung an Übung 1+2

Eigene Aufgabe wählen:

exercises/ex-1/ oder eigener Code — markiert die Stelle, an der ihr etwas ändern, verstehen oder verbessern wollt · Agent/Copilot öffnen

SCHRITT 1

Wählt 2–3 Techniken und eure Aufgabe — Refactoring, Tests, Bug, Erklärung, Review …

SCHRITT 2

Führt den Starter-Prompt je Technik aus. Passt ihn bei Bedarf an — Technik bleibt erkennbar.

SCHRITT 3

Vergleicht: Was liefert jede Technik anders? Welche passt am besten zu eurer Aufgabe?

Starter-Prompts als Vorlage — auf eure Aufgabe anpassen:

01

Plan-First

„Plan für [eure Aufgabe]. 6 Schritte. Kein Code. Risiken?"

02

Test-Driven

„Tests für [markierte Logik]. AAA, node:test. Happy, Edge, Error. Nur Tests."

03

Review-First

„Review [markierte Stelle]. Risiken? Fehlende Tests? Intent? Kein Code ändern."

04

Rubber-Duck

„Leeres Array → 4,99 statt 0. Ist das Bug oder Feature? Top-3 Hypothesen + Prüf-Steps."

05

Explain-Before

„Erkläre [markierte Stelle] in 5 Bullets: Input, Output, Edge Cases. Kein Code."

06

Spec-Driven

„3 Akzeptanzkriterien für [Aufgabe]. Dann Umsetzung."

07

Few-Shot

„[Aufgabe] im Stil von [Beispiel-Datei im Repo] — gleiches Pattern."

08

Red-Team

„Finde 3 Schwächen in [eurem Output/Code]. Security, Edge Cases, Lücken."

Debrief-Frage: Welche Technik hat den größten Unterschied gemacht? · Optional: besten Prompt im Team teilen

KI-Antworten sind Vorschläge, keine Freigaben

Die KI liefert einen Vorschlag — du entscheidest erst nach Review, ob er ins Repo gehört.

Schritt 1

KI-Vorschlag

Code, Diff, Erklärung — schnell, aber nicht geprüft

→
Schritt 2 · Pflicht

Du reviewst

Schritt 3

Deine Freigabe

Commit, Merge, Deploy — erst nach Review
